
# PROMPT FOR COMMENTS (v0.4.0)
- Comments for source codes.





---





Frens, you are a code explaining master. I need your help adding comments to the existing source files.



## 1. Rules
01. Add comments to the source files in `src/01_webapp/`.
	01-01. Add comments only to `*.ts` files.
	01-02. Detailed comments must always be added to `interface`.
02. At the top of each file, add a comment like JavaDoc.
	02-01. It must provide an overall explanation of how the file is used.
	02-02. The source file comment starts after the import block, with 2 blank lines between the final `import` and the comment.
	02-03. If the source file comment directly documents the next single code block, do not add a blank line after the comment.
	02-04. If the source file comment describes the whole file/module and the file has multiple top-level code blocks, add 2 blank lines after the comment.
03. Each function must include a brief description of what it does.
	03-01. If the function is simple, then a comment is not necessary.
04. Complex logic must include inline comment lines.
05. Use easy and simple English.
06. It must be instantly understandable at a glance for dev frens.
	06-01. Remember that comments are for humans.




## 2. Example

```ts

import { MyDependency } from 'mydependency';


/**
 * # FOO
 * - This comment describes the whole file/module.
 * - Use 2 blank lines after it when multiple top-level code blocks follow.
 */


/**
 * This comment directly documents FooEvents, so no blank line after the comment.
 */
interface FooEvents {
	// Called when foo changes.
	onFooChange: () => void;
}


/**
 * This comment directly documents Foo, so no blank line after the comment.
 */
export function Foo(): void {

	/**
	 * This comment directly documents doFoo, so no blank line after the comment.
	 */
	function doFoo(): number {
		// Explain complex logic with inline comments.
		<logic here..>
	
		// Use multiple lines if the logic needs it.
		<logic there..>
	}
}

```

01. The source file comment:
	01-01. Title: Use `#` and upper-case letters, just like markdown.
	01-02. Explanation: Use `-` as a list item, just like markdown.
	01-03. It starts after `import`, with 2 blank lines between the final `import` and the comment.
	01-04. If the comment covers one immediately following code block, do not add a blank line after the comment.
	01-05. If the comment covers the whole file/module and several top-level code blocks follow, add 2 blank lines after the comment.
02. The function comment:
	02-01. Just explanation, no title needed.
	02-02. If the function is very large, use markdown-like commenting style.
03. The logic comment:
	03-01. Use `//`.
	03-02. If the logic is complex, use multiple lines of `//`.
04. Inside of an interface, use `//`.
	04-01. `/**` & `*/` are for a big category.
